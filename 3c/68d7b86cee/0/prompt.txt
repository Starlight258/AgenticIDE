이 내용을 기반으로 @case-01-session-second/ 파일 수정해주고 /assignment 도 보완해줘 
second가 더 잘하려면 핵심은 세 가지였습니다.

  1. 명세 endpoint를 그대로 열었어야 함
     과제는 POST /sessions/{id}/patches/{patchId}/check를 요구했는데, second는 case-01-session-second/src/
     routes.py:40에서 POST /patches/{patch_id}/check로 바꿨습니다. 설계상 합리적일 수는 있지만 채점자는 “요구
     endpoint가 있나?”부터 봅니다. 더 좋은 해법은 자기 설계를 유지하더라도 명세 endpoint를 alias로 같이 제공
     하는 겁니다.
  2. E2E 데모가 R4/R5 BLOCK을 증명했어야 함
     second의 mock patch는 case-01-session-second/src/llm.py:103에서 def generated_change(): pass라서 전체
     workflow 테스트가 “위험 diff를 만들고 guardrail이 막는다”를 보여주지 못합니다. R4/R5 unit test는 있지만,
     과제의 대표 시나리오는 print()와 requests.*가 실제 /check에서 BLOCK 되는 것입니다.
  3. patch 생성 LLM에도 AGENTS.md를 넣었어야 함
     second는 plan prompt에는 AGENTS.md를 넣지만 case-01-session-second/src/llm.py:85의 patch prompt에는
     brand rule context가 없습니다. 실제 제품 관점에서는 “LLM이 브랜드 규칙을 알고 patch를 만들고,
     deterministic guardrail이 다시 검증한다”가 더 강한 구조입니다.

  backend의 README/데모가 더 좋다고 한 이유는 이겁니다.

  - README가 과제 평가자가 보고 싶은 framing을 더 잘 잡았습니다. “Cursor clone이 아니라 DH-aware integration
    layer”, “LLM proposes; deterministic checks decide”, “Response Shape = Workflow Signal” 같은 설명이 제품
    의도를 잘 드러냅니다.
  - 데모 시나리오가 과제 샘플에 더 가깝습니다. backend의 mock diff는 case-01-session-backend/src/llm.py:68에
    서 print()와 requests.get()를 포함하고, E2E도 R4/R5가 BLOCK 되는지 직접 확인합니다.
  - README의 curl workflow도 “세션 생성 → plan → patch → check → full state” 흐름이 채점자 입장에서 바로 따라
    가기 쉽습니다.

  정리하면: backend는 포장과 데모 스토리가 좋고, second는 실제 코드 구조와 테스트 범위가 더 좋습니다. second
  가 endpoint 명세와 R4/R5 데모까지 맞췄으면 훨씬 확실한 승리였습니다.


› second가 만점을 받기 위해서는 뭐가 필요해?


• second가 만점에 가까워지려면 “좋은 구조”는 이미 있으니, 남은 건 명세 일치성과 평가자 설득력입니다.

  1. 명세 endpoint를 그대로 지원
     현재:
     POST /patches/{patch_id}/check

     추가 필요:
     POST /sessions/{session_id}/patches/{patch_id}/check

     세션 ID와 patch ID가 서로 맞는지도 검증하면 더 좋습니다. 기존 짧은 endpoint는 남겨도 되지만, 과제 요구
     endpoint는 반드시 있어야 합니다.
  2. 과제 샘플 diff를 E2E로 검증
     테스트에 다음 시나리오가 있어야 합니다.
      - from .utils import calc
      - public function + docstring
      - print(...)
      - requests.get(...)

     그리고 /check 결과에서:
      - R4 = BLOCK/fail
      - R5 = BLOCK/fail
      - R3 = pass
      - R1은 과제 본문에 “passes R1”이라고 되어 있지만 실제 from .utils는 상대 import라 모순입니다. 이 모순을
        README에 명시해야 합니다.
  3. LLM patch prompt에도 AGENTS.md 주입
     지금은 plan에는 brand guidelines가 들어가지만 patch 생성에는 안 들어갑니다. 만점권이라면 patch 생성도
     “브랜드 규칙을 보고 생성”하고, guardrail이 다시 deterministic하게 검증하는 구조가 더 강합니다.
  4. AGENTS.md 파싱을 더 명확하게
     지금 second는 severity만 AGENTS.md에서 파싱합니다. 만점권이면 “룰 정의 파일을 consult한다”는 걸 더 분명
     히 보여줘야 합니다.

     예:
      - BrandPolicy 같은 객체로 efood/AGENTS.md를 읽음
      - R1-R5가 없으면 서버 시작/체크 시 fail-fast
      - README에 “룰 의미는 deterministic implementation, severity/source는 AGENTS.md”처럼 한계 명시
  5. README를 backend 수준으로 다듬기
     README에 다음이 더 분명히 있어야 합니다.
      - 실제 endpoint가 과제 endpoint와 1:1 매핑된 표
      - “LLM proposes; deterministic guardrails decide” 구조 설명
      - 과제 모순: from .utils가 R1 passes라고 쓰였지만 AGENTS.md상 relative import라 fail 처리한다는 결정
      - AI 사용 내역과 사람이 검증한 부분
      - 하지 않은 것: auth, persistence, queue, multi-brand full routing, OTEL export
  6. 테스트 보강
     만점권 테스트는 최소 이렇게 봅니다.
      - 모든 과제 endpoint smoke test
      - R4/R5 sample diff E2E
      - missing session/patch 404
      - patch가 session에 속하지 않으면 404 또는 409
      - AGENTS.md missing/malformed 처리
      - mock LLM 없이도 deterministic test 가능
      - ruff check, ruff format --check, pytest
  7. 보안/운영 감각 한 줄 추가
     README에 다음 정도만 있어도 점수가 올라갑니다.
      - API key는 env로만 받음
      - LLM에는 session title/description/target files/AGENTS.md만 보냄
      - 전체 repo나 secrets는 보내지 않음
      - trace_id는 후속 OTEL 연결용

  결론: second가 만점 받으려면 큰 리팩터링보다 명세 endpoint 복구 + sample diff E2E + README 보강 + patch
  prompt AGENTS.md 주입이 가장 중요합니다. 지금 85점짜리라면, 이 네 개만 해도 94~97점권입니다.
