## 1. 어떤 인재인가 — 신규 팀 + IC1의 특수성

### 1-1. 한 줄

> **"AI를 통제하면서 작은 도구를 끝까지 만들어본 사람, ambiguity 앞에서 자기 결정 내리는 사람"**

신규 팀의 IC1은 "지시 기다림"이 아니라 작은 owning 단위 끝까지 책임지는 게 디폴트. 시니어 1-2명이 큰 방향 잡고, IC1은 그 안의 작은 기능 1-2개를 처음부터 끝까지 끌고 간다. Senior 마인드도 Junior 마인드도 아닌 **"미니 owner"**.

### 1-2. JD 핵심 책임 5개 → 시그널 매핑

| # | 시그널 | JD 키워드 | IC1 기대치 |
|---|---|---|---|
| 1 | **AI 통제력** | "guided feature generation" / "Smart Guardrails" | LLM 호출 + 결과 검증 루프를 1번 이상 직접 짜본 흔적 |
| 2 | **시스템 통합 감각** | "Contextual Intelligence" / distributed ecosystem | 외부 데이터·API·MCP-류를 본인이 wrap해서 LLM에 먹여본 경험 |
| 3 | **결정 + 문서화** | "Strategic Customization" / "AGENTS.md" | spec 애매한 부분에 assumption 박고 trade-off 적는 습관 |
| 4 | **개발자 공감** | "Developer Empathy" | 본인이 매일 쓰는 도구의 마찰을 식별·개선해본 경험 |
| 5 | **소통·도입** | "Evangelism and Training" | 작은 워크숍·문서·동료 페어링이라도 운영해본 흔적 |

### 1-3. 안티패턴 (즉시 탈락 신호)

- **AI 생성 코드를 검증 없이 붙이는 후보** — dead code, 의미 없는 abstraction, 일관성 없는 네이밍 → "AI 도구 메이커"가 본인이 통제 못 하는 도구를 만들 리 없음
- **알고리즘만 잘 푸는 후보** — spec disambiguation·README·trade-off 설명 못 하면 신규 팀에서 부담
- **방향 받기만 기다리는 후보** — ambiguity 앞에서 멈추고 질문만 던지는 사람
- **Standard 저항형** — AGENTS.md·guardrail·표준화에 시큰둥. 우리가 만드는 건 1인용이 아니라 1,000인용
- **Evangelism 0** — 코드만 짜고 다른 팀과 일 못 하는 사람. eFood·Glovo·Talabat 협업이 디폴트

### 1-4. 신규 팀이라서 더 중요한 것

- **Ambiguity tolerance** — 우리도 spec 미정. 같이 만들어가는 사람
- **Transferable thinking** — 다른 도메인(데이터 파이프라인, MCP, LLM 오케스트레이션)에서 비슷한 패턴 풀어본 사람 → 외부 채용의 정당성
- **Bootstrap mindset** — README가 없는 환경에서 본인이 첫 README를 쓰는 사람

## 2. Case Study 설계 원칙

### 2-1. 무엇을 평가하지 **않을** 것인가 (먼저 깎기)

- ❌ 알고리즘 정답성 — LeetCode 아님
- ❌ 시간 안에 다 끝냈는가 — 4시간이라 못 끝냄을 가정
- ❌ 라이브러리 지식 자랑 — Cursor 잘 쓰는 게 핵심 X
- ❌ AI 안 쓰고 손코딩 — 오히려 감점. "AI Leverage" 명시

### 2-2. 5축 루브릭 (25점 만점) — v2 (외부 calibration 반영)

| 축 | 비중 | 보는 신호 |
|---|---|---|
| **Spec Disambiguation** | 20% (5점) | 애매한 요구사항을 assumption으로 정리·README 명시·trade-off 1개+ |
| **AI Leverage 통제** | 20% (5점) | LLM 어디 썼고 / 검증 어떻게 했고 / 직접 짠 부분이 README·commit에서 추적 가능 |
| **System Sense** | 25% (6점) | 비결정적 출력에 가드레일 1개+, 에러 처리, "100배 입력이면?" 답 가능, **secrets·PII·internal code 보호 의식** |
| **Code/Doc Quality** | 25% (6점) | **돌아가는 코드가 우선**. readable, naming, dead code 없음, working tests, README 5섹션 |
| **Empathy** | 10% (3점) | README가 다음 엔지니어 시점, 실행 명령어 진짜 돌아가나, multi-brand·multi-team 맥락 의식 |

> **v2 변경 이유**: AI Leverage 25→20, Code/Doc 20→25. "AI를 썼는가"보다 **"작고 안전하게 완성했는가"**가 더 강한 시그널. README만 화려하고 코드 안 돌아가면 즉시 bubble.

**컷오프**:
- 20+ → Strong yes
- 15~19 → Discussion에서 결정
- 15 미만 → No (특히 안티패턴 1개라도 보이면 즉시)

### 2-2b. Security/Privacy 가드레일 (System Sense 안의 핵심 sub-축)

DH 같은 글로벌 회사에서 Agentic IDE는 **내부 코드·diff·문서·Slack**을 LLM에 흘려보내는 도구. 보안 의식 없으면 즉시 감점.

| 신호 | 보는 것 |
|---|---|
| **Secrets 처리** | API key·token이 코드/로그/output에 노출되지 않음. `.env`·환경변수·secret detection 1개+ |
| **PII 마스킹** | 사용자 데이터·이메일·이름이 LLM 프롬프트에 들어가기 전 sanitize |
| **Internal code 전송 통제** | 외부 LLM에 전체 코드 베이스 전송 X. 필요한 컨텍스트만 골라서 |
| **Audit log** | LLM에 무엇을 보냈고 무엇을 받았는지 1줄이라도 기록 |

→ 후보가 README에 위 4개 중 1개라도 명시하면 +1점 (System Sense 안에서)

### 2-3. 형식 — 의도된 제약

- **4시간 timed** — 못 끝내는 게 디폴트. 무엇을 끊고 무엇을 살렸는지가 시그널
- **GitHub repo 제출** — commit 잘게 vs 통째로 1개 → 작업 흐름 가시성 차이
- **AI 활용 노골적으로 허용** — "어떻게 통제했는가"가 평가축임을 명시
- **2~3개 의도된 ambiguity** — spec에 일부러 빈칸 → assumption 노트 강제
- **README는 평가의 50%** — 코드만 안 봄을 미리 안내

### 2-4. 추천 시나리오 — AGENTS.md Generator + Lint Guardrail

> **과제 한 줄**: 회사 코딩 가이드라인 마크다운을 입력받아 (1) `AGENTS.md`를 자동 생성하고, (2) 임의의 PR diff가 그 가이드라인을 위반하는지 감지하는 CLI 도구를 만드세요.

**왜 이 시나리오인가** — JD 5개 책임을 한 과제에서 다 건드림:

| JD 책임 | 시나리오에서 어떻게 측정 |
|---|---|
| Strategic Customization | AGENTS.md 생성 = "Engineering Manifesto 통합" 그 자체 |
| Smart Guardrails | PR diff 위반 감지 = 비결정적 출력에 결정적 검증 |
| Contextual Intelligence | 가이드라인 .md → 구조화된 컨텍스트 변환 |
| Driving Productivity | CLI = 다른 엔지니어가 매일 쓸 도구 (Developer Empathy) |
| Adoption | README가 동료 엔지니어를 설득해야 함 |

**의도된 ambiguity 3개**:
1. 입력 가이드라인이 **모순**될 때 어떻게? (예: "함수는 짧게" vs "에러 처리 풍부히")
2. 위반 감지를 **LLM으로** 할까, **deterministic rule**로 할까? — 양쪽 trade-off가 있게 설계
3. 출력 AGENTS.md의 **우선순위 정렬 기준**? — 강제성 / 등장 빈도 / 위반 빈도

**입력 자료** (지원자에게 제공):
- `coding-guidelines.md` (~30 rules, 일부 모순 박힌 채)
- `pr-diff-{1..5}.patch` (위반 3개 / 비위반 2개)
- `README-template.md` (assumption / trade-off / AI leverage 섹션 강제)

**시간 분배 가이드** (제출 메일에 명시):
- 0-30분: spec 분해 + assumption 박기
- 30분-3h: 구현
- 3h-3.5h: 테스트
- 3.5h-4h: README 마무리

### 2-5. Discussion 단계 — 30분 / 4축 질문

설계 의도: "코드 잘 짜는 사람" vs "결정을 설명할 수 있는 사람"을 가르는 단계.

| 축 | 질문 예시 | 보는 시그널 |
|---|---|---|
| 결정 reasoning | "LLM에 맡긴 부분 vs deterministic 룰로 처리한 부분의 경계 어떻게 정했어요?" | trade-off 명시 능력 |
| AI 통제 | "LLM 출력 그대로 받았어요? 검증 어떻게 했어요?" | 비결정적 시스템 사고 |
| 스케일 | "1,000개 가이드라인 + 1만 PR이면 뭐가 깨져요?" | "100배 사고" |
| 도입 | "이 도구를 다른 팀에 깔려면 첫 한 주에 뭐부터?" | Evangelism |

각 질문 30초 침묵 = 감점. 모르는 건 솔직하게 + "내 직관은 X"가 정답.

## 3. 운영 체크리스트 (평가자 측)

- [ ] LLM API key 안내 — 자체 충전 vs 회사 제공 정책
- [ ] 4시간 시계는 candidate self-report (정직성 신뢰)
- [ ] 평가자 2명 이상 독립 채점 후 calibration session
- [ ] Discussion은 코드 보면서 pair-review 형식
- [ ] 합격선 자문: **"이 사람과 같이 첫 3개월 신규 팀 만들 수 있나?"**

## 4. 한 줄 결론

> **"우리가 뽑는 건 Cursor를 잘 쓰는 사람이 아니라, Cursor가 만들어준 코드를 통제할 수 있는 사람이다."**
> 
> 신규 팀 IC1은 "지시 받는 사람"이 아니라 "작은 도구를 끝까지 owning하는 사람". 그래서 4시간 case study는 "정답이 있는 문제"가 아니라 **"결정의 흔적이 남는 문제"**여야 한다.

## 5. 본인(지원자 시점) 활용 가이드

이 문서를 거꾸로 읽으면 **케이스 스터디에서 점수 따는 행동 리스트**:

1. **README를 코드보다 먼저 신경 써라** — 평가 50%
2. **assumption 섹션을 의도적으로 만들어라** — 애매한 spec이 없어 보여도 무조건 박기
3. **AI Leverage 섹션을 길게 써라** — 어디 썼고 / 어떻게 검증했고 / 직접 짠 게 어디인지
4. **deterministic vs non-deterministic 경계를 의식해서 설계해라** — guardrail 시그널
5. **"Not Done" 섹션을 부끄러워하지 마라** — 자기 비판 = 시스템 사고 신호
6. **commit을 잘게 나눠라** — 작업 흐름 가시성
7. **실행 명령어가 진짜 돌아가는지 마지막 5분 확인** — Empathy 시그널
