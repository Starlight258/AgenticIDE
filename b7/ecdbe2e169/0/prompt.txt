설계시에 이거 고려해줘 
핵심 설계 결정 정리

1. Patch는 “적용된 코드”가 아니라 candidate change

결정

Patch = merge 전 후보 diff

실제 코드베이스 상태를 변경하지 않음.

⸻

이유

Agentic IDE workflow는 iterative함.

Patch 1 -> 실패
Patch 2 -> 수정
Patch 3 -> 최종 후보

과거 patch는 draft/history 역할.

중요한 건:

현재 candidate patch가 안전한가?

⸻

Trade-off

장점    단점
모델 단순    실제 git merge state 없음
readiness 계산 쉬움    branch/rebase 문제 미모델링
AI iterative workflow와 잘 맞음    patch dependency 추적 안 함

⸻

2. Step readiness는 latest patch 기준

결정

latest checked patch 기준 readiness 계산

최신 patch에 BLOCK violation 있으면 NOT_READY.

⸻

이유

과거 patch 실패는 수정될 수 있음.

historical failure
≠
current merge safety

⸻

Readiness 정책

BLOCK 존재 → NOT_READY
WARNING만 존재 → READY

⸻

Trade-off

장점    단점
현실적 iterative flow    과거 실패 무시
merge candidate 중심 사고    patch lineage 없음
구현 단순    selected patch 개념 없음

⸻

3. Guardrail 결과는 immutable check record로 저장

결정

CheckResult를 patch에 저장

readiness 계산 시 diff를 재분석하지 않음.

⸻

이유

검사는 하나의 verification event임.

Patch content
→ Check execution
→ Stored findings

상태 재계산보다 audit trail 유지가 중요.

⸻

Trade-off

장점    단점
deterministic    stale result 가능
audit 가능    patch 변경 시 재체크 필요
readiness 계산 빠름    캐시 invalidation 필요 가능

⸻

4. LLM output과 execution evidence 분리

결정

리소스    역할
Plan / Patch    LLM-produced
CheckResult    system-computed
TestRun    human/automation recorded

⸻

이유

AI claim
≠
verified execution

AI가 “tests passed”라고 주장해도 실제 CI는 실패 가능.

⸻

Trade-off

장점    단점
trust boundary 명확    모델 수 증가
auditability 향상    구현 조금 복잡
현실적인 CI workflow    nested state 깊어짐

⸻

5. TestRun은 session-level resource

결정

endpoint:

POST /sessions/{id}/test-runs

TestRun은 patch_ids 배열로 여러 patch 연결 가능.

⸻

이유

실제 테스트는 보통 여러 변경을 함께 실행함.

pytest 전체 실행
integration suite 실행

⸻

Trade-off

장점    단점
현실적    patch별 isolation 약함
batch execution 가능    aggregation 약간 복잡
endpoint와 자연스럽게 맞음    traceability 추가 필요

⸻

6. Severity ladder

결정

BLOCK
WARNING
INFO

⸻

정책

Severity    의미
BLOCK    merge 불가
WARNING    review 필요
INFO    참고용

⸻

예시

Rule    Severity
G1 Decimal violation    BLOCK
G2 hardcoded URL    BLOCK
G3 DB session violation    BLOCK
G4 trace id missing    WARNING
G5 missing docstring    WARNING

⸻

이유

glovo는 money-sensitive system.

correctness/security violations are merge blockers

README에 넣을 핵심 문장

Architecture intent

The service models an AI-assisted coding workflow rather than a Git repository.
Patches are treated as candidate changes generated by the LLM and are evaluated independently before merge readiness.

⸻

Readiness rationale

Step readiness represents merge safety for the current candidate patch, not historical patch quality.
Earlier patches may fail and later patches may fix those violations, so readiness is computed from the latest checked patch only.

⸻

Trust boundary

The system explicitly separates:
- LLM-produced artifacts (Plan, Patch)
- backend-computed validations (CheckResult, Readiness)
- reviewer/automation evidence (TestRun)
This prevents the backend from trusting the LLM's claim that a patch works.

⸻

Not done (의도적 생략)

추천

- No real git application or merge execution
- No branch/rebase model
- No asynchronous job queue
- No real test execution sandbox
- No patch dependency graph
- No auth/authz
- No persistence beyond local storage

⸻

면접/README에서 좋아할 포인트

이 과제는 사실:

AI workflow state machine

설계 문제임.

좋은 포인트는:

* current candidate vs historical draft 구분
* generated state vs verified state 분리
* merge safety invariant 정의
* trust boundary 모델링

이 네 개야.
